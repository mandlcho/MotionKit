# Animator Workflow Automation — Design Spec

**Date**: 2026-05-17
**Status**: Draft for review
**Author**: TD (owner)

## Problem

The game has a continuously-growing roster of characters with cloth-bearing attachments — skirts, tails, long hair, sleeves — that today require per-animation manual labor:

1. **Keyframing the attachment's root joint** in 3dsmax so the attachment doesn't clip the body during motion. The animator judges this frame-by-frame from the body pose.
2. **Setting up and running cloth sim** in 3dsmax with character-specific configuration, then baking the result onto bones.

The result is then exported to UE, where Kawaii Physics adds runtime wobble.

The cloth sim itself produces visual quality (mesh-level clipping prevention, directional draping) that runtime bone-spring systems like KP cannot reproduce. Removing the cloth sim is not on the table.

At current scale (8 body type templates, many characters per type, large animation sets per character, continuous content authoring), per-animation manual labor does not scale. The fix is to move human time from per-animation to per-character, with everything in between automated.

## Goals

- Per-animation manual work for cloth attachments goes to **zero**. Animator time per character is bounded (one-time setup) and amortizes across every animation that character ever receives.
- Keep the offline 3dsmax cloth sim — its visual output is load-bearing.
- All secondary motion is baked offline; runtime cost is unchanged (KP layer in UE only).
- Per-character setup is animator-friendly: visual editor in 3dsmax, no JSON hand-editing.
- Support all attachment types uniformly: skirt, tail, hair (long chains), sleeves.
- Pipeline is single-tool (3dsmax). No round-trip to UE for the offline work.

## Non-Goals

- Replace Kawaii Physics. KP layer in UE remains as-is.
- Replace cloth sim with Chaos Cloth or any runtime cloth solver. Bone-driven, offline-baked only.
- Round-trip to UE for offline processing. UE only does runtime.
- Handle every novel pose the curve authors never anticipated. Phase 1 explicitly trades extrapolation robustness for simplicity; if production data later shows this is needed, a Phase 3 collision safety net can slot in.

## Architecture

```
Animator authors body animation in 3dsmax (existing workflow)
        │
        ▼
[Auto-Root]  ◄── reads resolved character config (driver curves)
        │
        ▼
Corrected animation: attachment roots auto-keyframed
        │
        ▼
[Batch Cloth Sim Runner]  ◄── reads resolved character config (cloth material, body capsules)
        │
        ▼
Baked FBX  →  Export to UE  →  KP runtime (unchanged)
```

All tools live in 3dsmax. Pipeline is offline batch.

The **auto-root is driver-driven**, not collision-driven: each attachment root reacts to specified driver bones via authored response curves. When the left thigh rotates, the skirt root rotates according to the curve. This is the 3dsmax-native equivalent of UE's Spherical Pose Reader and maps directly onto **Reaction Manager**.

## Data Model

Three JSON artifact types, with two-level inheritance (body type → character).

### 1. Material library — `cloth_materials.json`

Shared physics presets for cloth sim. TD-owned, version-controlled.

```json
{
  "materials": {
    "soft_pleated_cotton": { "stiffness": 0.30, "bend": 0.20, "damping": 0.55, "airResistance": 0.10, "mass": 1.0 },
    "heavy_leather":       { "stiffness": 0.85, "bend": 0.75, "damping": 0.40, "airResistance": 0.02, "mass": 2.4 },
    "flowing_silk":        { "stiffness": 0.12, "bend": 0.08, "damping": 0.30, "airResistance": 0.15, "mass": 0.6 }
  }
}
```

### 2. Body type templates — `body_types/*.json`

Eight templates (`AdultMale`, `AdultFemale`, `TeenMale`, `TeenFemale`, `Child`, etc. — exact list TBD by team). TD-owned. Each defines: skeleton expectations, default collision sets used by the cloth sim, and default driver relationships per attachment type.

```json
{
  "id": "AdultMale",
  "skeletonTemplate": "biped_v3",

  "collisionSets": {
    "legs":      { "bones": ["LeftUpLeg","RightUpLeg","LeftLeg","RightLeg"], "scaleFactor": 0.9 },
    "hips":      { "bones": ["Hip","Pelvis"], "scaleFactor": 0.95 },
    "torso":     { "bones": ["Spine_01","Spine_02","Spine_03"], "scaleFactor": 0.92 },
    "shoulders": { "bones": ["LeftShoulder","RightShoulder","LeftArm","RightArm"], "scaleFactor": 0.9 }
  },

  "attachmentDriverDefaults": {
    "skirt": {
      "drivers": [
        {
          "driverBone": "LeftUpLeg",
          "driverAxis": "rotation_x",
          "responseCurve": [
            { "driverValue": -90, "rootDelta": { "rotX":  25, "rotY":  5, "rotZ": 0 } },
            { "driverValue":   0, "rootDelta": { "rotX":   0, "rotY":  0, "rotZ": 0 } },
            { "driverValue":  90, "rootDelta": { "rotX": -10, "rotY": -5, "rotZ": 0 } }
          ]
        },
        { "driverBone": "RightUpLeg", "driverAxis": "rotation_x", "responseCurve": [/* mirrored */] }
      ],
      "constraints": { "maxRotationDegrees": 35 },
      "extrapolation": "clamp",
      "temporalSmoothingFrames": 3
    },
    "tail":     { "drivers": [/*...*/], "constraints": { "maxRotationDegrees": 50 } },
    "hairLong": { "drivers": [/*...*/], "constraints": { "maxRotationDegrees": 60 } },
    "sleeve":   { "drivers": [/*...*/], "constraints": { "maxRotationDegrees": 45 } }
  },

  "qaThresholds": {
    "maxVertexVelocity": 50,
    "maxSelfIntersectionsPerFrame": 5,
    "maxRootRotationJumpDegrees": 10
  }
}
```

### 3. Character preset — `characters/*.json`

One per character. Animator-edited via visual panel. Sparse — only declares what's specific or overridden.

```json
{
  "id": "AdultMale_Rex",
  "extends": "AdultMale",

  "collisionSetOverrides": {
    "hips": { "scaleFactor": 1.05 }
  },

  "attachments": [
    {
      "name": "skirt",
      "rootBone": "Bip_Skirt_Root",
      "type": "skirt",
      "clothSim": { "material": "soft_pleated_cotton" }
    },
    {
      "name": "ponytail",
      "rootBone": "Bip_Hair_Tail_Root",
      "type": "hairLong",
      "clothSim": null,
      "driverOverrides": [
        /* optional: override the body type's default curve for this character */
      ]
    }
  ]
}
```

### Resolved config

At load time, the three artifacts merge into a single dense **resolved config** that all tools consume: `materials + body_type + character_preset → resolved`. Character preset wins on conflict, body type provides defaults, material library provides physics constants. The merge is a single utility function used by every tool — never duplicated.

### Validation (enforced at load)

- `extends` must reference a known body type.
- Each attachment's `type` must exist in the body type's `attachmentDriverDefaults`.
- All referenced bones must exist in the scene's skeleton.
- All referenced materials must exist in the material library.
- Response curve key points must be monotonically increasing in `driverValue`.

Validation runs both in the Driver Setup panel at save time *and* in the batch runner at load time. Malformed presets produce clear actionable error messages, never silent failures.

## Components

### Auto-Root

Per-frame curve evaluator. For each registered attachment in the resolved character config:

1. Read each driver bone's current rotation on its specified axis.
2. Evaluate the response curve at that driver value via linear interpolation between key points; clamp outside authored range (Phase 1 behavior).
3. Sum the `rootDelta` contributions across all drivers for this attachment.
4. Clamp the total against `maxRotationDegrees`.
5. Write the result as a keyframe on the attachment's `rootBone`.
6. Apply moving-average temporal smoothing across the configured frame window (default 3) to eliminate sub-frame jitter from upstream driver bone noise.

Inputs: source FBX (body animation), resolved character config.
Output: FBX with attachment root bones keyframed.

Implementation note: prefer wrapping 3dsmax's built-in **Reaction Manager** as the curve evaluation engine. Animators may already be familiar with it; less code to maintain than a custom evaluator. Validate during Phase 1 prototype that Reaction Manager's interpolation behavior matches what we want.

### Driver Setup Tool

Visual panel in 3dsmax (Python-in-Max with PySide rollout, or MaxScript rollout — language decision pending). Animators use this to author per-character driver curves where the body-type defaults aren't sufficient.

UI:
- Character header: load/save preset, body type dropdown (for new characters)
- Tab per declared attachment
- Per driver: bone picker, axis selector, curve editor (graph view + editable key-point table)
- **"Preview on current frame"** button — runs the Auto-Root evaluator on the active scene's current frame, applies the resulting keyframe so the animator immediately sees the response
- **"Preview across range"** button — runs Auto-Root over a scrub range, lets the animator see the response evolve across motion
- **Validation on save** — referenced bones exist, curves are monotonic, material reference is valid, etc.

The panel is the only place animators touch presets. JSON is the storage format, not the editing surface.

### Batch Cloth Sim Runner

Processes a folder (or queue) of FBXs (output from Auto-Root) through 3dsmax cloth sim.

Per FBX:
1. Open in 3dsmax (headless mode if licensable; otherwise standard).
2. Load resolved character config.
3. Apply cloth modifier to the registered cloth attachments using the material's physics params.
4. Auto-generate body collision capsules from skinning weights × character's `collisionSets` definitions, applying `scaleFactor` (uses the Capsule Auto-Generator output, cached per character).
5. Simulate.
6. Bake cloth mesh deformation back to attachment child bones (skin-wrap or vertex-driven bone fitting — use the existing project technique).
7. Export FBX.
8. Run QA checks against `qaThresholds`; append findings to the batch report.

Initial implementation is single-process per machine. Each FBX is independent — trivially parallelizable across machines in Phase 2 via a job queue (e.g., Deadline, custom Python worker pool).

### Capsule Auto-Generator

Utility, runs once per character (or whenever the character's mesh changes). For each bone listed in the character's resolved `collisionSets`:

1. Find vertices with skinning weight > 0.5 on that bone.
2. Compute a bounding capsule via PCA — longest principal axis becomes the capsule axis; perpendicular spread becomes the radius.
3. Apply the resolved `scaleFactor` (body type default, possibly overridden per character).
4. Persist as scene-bound primitive objects parented to bones AND export a debug visualization (overlay in viewport) for TD review.
5. Cache the capsule set as part of the character's setup data; need not rerun per animation.

If no skinning weights pass the threshold for a listed bone, the auto-generator surfaces this clearly rather than producing a malformed capsule.

### QA Dashboard

Phase 1: structured output, no heavyweight UI.

- Each batch run writes a JSON report: per-FBX status (success/flagged/failed), per-frame anomalies with frame numbers, baked-bone curve statistics.
- A 3dsmax rollout panel in Phase 2 reads the report and lists flagged FBXs with one-click "open this FBX at frame X" for TD review.

Anomaly signals (Phase 1):
- `maxVertexVelocity` exceeded on a frame
- `maxSelfIntersectionsPerFrame` exceeded
- Root rotation discontinuity exceeded `maxRootRotationJumpDegrees` frame-over-frame
- Sim non-convergence (cloth solver reports divergence)

Goal: TD review queue stays small (target ~5–10% of total animations). The rest go directly through to export.

## Data Flow

### Per-character setup (one-time, ~1–2 hours per character)

1. Animator creates new character preset via the Driver Setup Tool, picks body type from dropdown.
2. The tool pre-fills effective config from body type defaults; resolved view shows what's inherited vs character-specific.
3. Animator declares attachments — names them, points them at the right bones in the rig.
4. Capsule Auto-Generator runs against the character's skinned mesh. TD glances at the result, nudges any obvious misses (~15 min).
5. Animator scrubs a representative animation; if body-type-default driver curves look wrong for this character (e.g., heavyset character's skirt needs different response than the lean default), opens the per-attachment curve editor and adds an override.
6. Save preset.

### Per animation (zero human time, batch)

1. Animator authors body animation, drops the FBX into the input folder.
2. Auto-Root reads the corresponding character's resolved config, processes the FBX, writes a corrected FBX.
3. Batch Cloth Sim Runner picks up the corrected FBX, processes through cloth sim with the resolved cloth material + capsule set, exports baked FBX.
4. QA report updated with this FBX's result.
5. Flagged FBXs surface in the review queue; non-flagged go straight to UE import.

## Phased Delivery

### Phase 1 — MVP (target: build and ship for 1 body type + ~5 characters)

- Material library (initial 5–10 materials)
- 1 body type template (start with the most common one)
- Auto-Root with linear interpolation + clamp extrapolation, wrapping Reaction Manager
- Driver Setup panel (basic version, no animations preview)
- Batch Cloth Sim Runner (single-process)
- Capsule Auto-Generator
- Basic QA report (JSON only, no UI)
- Validation: schema validation on preset load

**Exit criteria**: 5 characters fully set up, ~50 animations batched end-to-end with zero per-animation human touch and visual quality matching the current manual pipeline.

### Phase 2 — Hardening (target: scale to all 8 body types)

- Remaining body type templates
- Animation-preview button in Driver Setup panel ("preview across range")
- QA dashboard UI (3dsmax rollout)
- Parallelism for batch runner (multi-machine job queue)
- Preset validation tool standalone (catch malformed presets at edit time without opening the panel)

### Phase 3 — Only if needed: Extrapolation safety net

Build only if Phase 1/2 production data shows extrapolation failures are a real production issue.

- Collision-based fallback that triggers when driver extrapolation would produce mesh clipping
- Hybrid: driver primary, collision deflects further if clipping detected
- Defense-in-depth: catches the long tail without complicating the common case

## Testing

- **Unit tests**: curve evaluator (deterministic math; trivial to test with input/output pairs), config resolver (merge logic), validation rules. Target high coverage; these are pure functions.
- **Integration tests**: golden-output tests — run a fixed character + animation through the full pipeline, byte-compare the baked FBX bones to a known-good output. Catches accidental regressions in the curve eval, smoothing, or sim setup.
- **Schema tests**: malformed presets produce expected error messages (each validation rule has a negative test).
- **Visual regression** (Phase 2): batch-render a fixed set of "calibration" animations across all body types; diff frames against a baseline. Not blocking for Phase 1.
- **Manual QA**: animator drives the Driver Setup panel through realistic workflows; surface UX issues early.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Driver curve authoring per body type takes longer than expected | Start with linear, rough curves; refine through production use. Body-type-level work amortizes across all characters of that type. |
| Reaction Manager can't express multi-driver blending we want | Prototype Phase 1 against it early; if blocked, fall back to a custom curve evaluator with the same external interface |
| Cloth sim wall-clock dominates total batch time | Parallelize Phase 2 across machines. Cloth sim per FBX is independent — trivial to fan out |
| Pose extrapolation produces broken root keyframes for unusual animations | Phase 1 clamps. Phase 3 adds collision safety net only if production data justifies it |
| Animators want to tune curves quickly | "Preview on current frame" + "Preview across range" buttons in the Driver Setup panel keep iteration loops tight |
| 3dsmax headless licensing limits batch throughput | Validate licensing model during Phase 1 prototype; budget licenses or fall back to interactive sessions on dedicated workstations |
